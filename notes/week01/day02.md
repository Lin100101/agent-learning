# Day 02 - Python 项目依赖管理与 Git 工作流实践

**日期：** 2026-07-08

## 今日目标

进一步熟悉现代 Python 项目的开发流程。

学习如何使用 `uv` 管理项目依赖，并结合 Git 完成代码版本管理。

---

## 今日学习内容

### 1. 认识现代 Python 项目配置

继续理解 Python 项目的核心文件：

```text
agent-learning/

├── pyproject.toml
├── uv.lock
├── .venv
└── .gitignore
```

了解到现代 Python 项目不再依赖传统的：

* requirements.txt
* 手动管理环境

而是通过项目配置文件和工具自动管理。

---

### 2. 使用 uv 管理依赖

学习了如何通过 `uv` 添加 Python 第三方库。

例如：

```bash
uv add requests
```

添加依赖后：

* `pyproject.toml` 会记录项目依赖；
* `uv.lock` 会自动更新。

这说明：

`pyproject.toml` 负责描述项目需要什么。

`uv.lock` 负责锁定具体安装什么版本。

---

### 3. 理解 uv.lock 的作用

进一步理解：

`uv.lock` 并不是开发者手动维护的配置文件，而是由 `uv` 自动生成。

它记录：

* 依赖包版本
* 依赖关系
* 环境解析结果

作用：

保证不同机器、不同开发者安装项目时拥有一致的环境。

---

### 4. Git 暂存区（Staging Area）

学习了 Git 的工作流程：

```text
修改文件

↓

git status

↓

git add

↓

暂存区

↓

git commit

↓

本地仓库
```

理解了：

`git add` 并不是提交代码。

它只是把修改放入一个"准备提交的区域"。

---

### 5. 理解 Git 暂存区的意义

认识到 Git 暂存区类似一个缓冲层。

它允许开发者：

* 检查即将提交的内容；
* 选择性提交文件；
* 避免错误提交。

例如：

```bash
git diff
```

查看工作区修改。

```bash
git diff --cached
```

查看已经暂存的修改。

---

发现最初创建的：

```text

```

结构不够符合每日学习记录。

因此调整为：

```text
week01/

```

让代码组织方式与学习进度保持一致。

---

## 今日最大收获

今天最大的收获不是学习某一个命令，而是理解了：

> 一个项目不仅需要代码，还需要依赖管理和版本管理。

Python 程序能够运行只是第一步。

一个可维护的项目还需要：

* 明确的依赖声明；
* 可复现的环境；
* 清晰的 Git 历史。

---

## 今日反思

以前我认为 Git 主要作用是"保存代码"。

今天理解到：

Git 更重要的作用是：

* 记录项目演变过程；
* 管理不同版本；
* 让开发者安全地修改代码。

同时，使用 `uv` 管理 Python 项目，让我认识到现代开发工具可以自动处理很多复杂工作。

开发者需要理解工具背后的机制，而不是只会执行命令。
